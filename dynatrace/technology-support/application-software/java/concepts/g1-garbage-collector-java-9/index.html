    G1 垃圾收集器 - Java 9

    Oracle 的 Java 9 Hotspot VM 附带垃圾优先（G1）GC 作为其默认垃圾收集器。这个 GC 首先在 Java 7 中引入，可以有效地同时处理非常大的堆。它也可以配置为不超过最大暂停时间。

    最先进的 GC 将堆分为新一代或旧一代对象。这主要是因为对实际 Java 应用程序的研究表明，90%以上的对象在第一次垃圾收集后就无法存活。经过几个收集周期存活下来的旧对象往往仍然存活，并且有98%的存活几率。Java GC 将年轻代对象进一步划分为幸存者空间和伊甸园空间。

    新分配的对象始终分配给伊甸园空间。一旦对象在其第一次垃圾收集中幸存下来，它就会移动到旧一代。这样做是为了：

    • 可以在新对象上使用运行时高效算法; 运行时仅取决于幸存对象的数量，但是浪费了堆大小的一半。

    • 老一代可以使用记忆效率高的算法; 运行时间取决于堆大小，但尽可能高效地使用可用内存。一堆这样的收集器看起来像这样：

    与大多数垃圾收集器相比，G1 有两大优势：

    • 它可以在不停止应用程序线程的情况下同时执行
    • 它使用非连续空间，使 G1 能够有效地处理非常大的堆

    由于它使用可用堆的方式，G1 可以同时收集年轻和老一代。而不是将堆分成3个空间，伊甸园，幸存者和旧空间，它将堆分成多个小区域，默认大小为 2MB。每个区域都分配给一个空间。

    将堆拆分成小区域使 G1 能够选择一小组区域来快速收集和完成。如果计划收集某个区域，则会将所有幸存的对象从收集的区域复制到未分配的区域。假设收集的区域属于伊甸园空间，则保存所有幸存物体的未分配区域将成为幸存区域。理想情况下，如果一个区域充满垃圾并且不包含单个幸存对象，则可以将其声明为“未分配”并且不对其进行任何操作。

    要收集整个堆，G1 可以选择要收集的任何数量或区域组合。为了优化收集时间，它总是选择充满或几乎充满垃圾的区域，从而最大限度地减少为后续分配释放堆空间所需的工作量。其他 GC 总是收集整代，因此它们的运行时复杂性取决于总堆大小。在 G1 的情况下，这取决于活动对象的数量，因为可以在不处理整个生成的情况下释放存储器。理想情况下，当堆足够大时，某些区域将始终充满垃圾，从而可以轻松收集它们。

    此外，G1 可以同时完成大部分工作。在 Java 世界中，我们已经了解了 Concurrent Mark＆Sweep GC（CMS）的并发集合。但是，CMS 只能同时收集旧代，并且仍然需要停止应用程序以收集年轻代。该过程发生在以下阶段：

    1. 初始标记：G1 仅在垃圾回收开始时停止应用程序，以便在恢复应用程序之前进行一些快速记账。
    2. 并发标记：在应用程序执行时，GC 遵循所有引用并标记生命对象。
    3. 最终标记：应用程序再次暂停，并进行最终清理。
    4. 疏散：选择和收集一些区域。

    由于疏散阶段很快，特别是在大堆的情况下，G1 在执行的应用程序的暂停时间方面通常优于其他 GC。

    缺点是 G1 在小堆中表现不佳，因此请考虑相关应用。如果需要，您可以通过设置-XX:+UseParallelOldGC标志返回默认收集器。如果 G1 的可用堆很少，您将在 GC 日志中看到 Full GCs。只有当 G1 确定不再可能使用通常的操作模式时，才会执行完整的 GC。在这种情况下，使用一种缓慢但内存效率高的算法来收集整个堆，这种算法使下一个收集变得更好。如果只有完整的 GC，可以考虑增加堆大小并选择不同的 GC。

    由于区域较小，G1 可以通过设置-XX:MaxGCPauseMillis=n来限制其最大暂停时间。然后 G1 会在估计可以立即收集的最大区域数量之前考虑先前的收集和检测到的垃圾量，而不会超出此限制。

    G1 还远不是一个实时收集器，但它的性能略好于其他受刚性堆结构限制的收集器。
