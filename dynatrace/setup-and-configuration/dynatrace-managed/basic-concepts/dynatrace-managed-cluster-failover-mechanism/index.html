    Dynatrace Managed 群集故障转移机制

    Dynatrace Managed 允许使用多个同等重要的节点构建的高可用性部署。虽然每个集群都包含一个主节点，但Dynatrace Managed 集群中的所有节点都运行相同的服务，并且能够担任主节点的角色。因此，不需要有关当前正在执行主节点角色的节点的信息，并且所有群集配置和维护都应通过 Cluster Management Console 完成。

    要实现最佳的故障转移部署，我们建议执行以下操作：

    • 冗余

    计划每个群集至少部署三个节点。在此类群集中，所有节点都会自动跨节点复制数据，因此除主要分片外还有两个副本。

    所有事件，用户会话和度量标准都以三个复制因子存储。Dynatrace 集群及其环境的整个配置存储在每个节点上。这意味着当一个数据节点完全不可用时，Dynatrace 可以完全继续运行，您需要运行多数仲裁：对于三个节点，两个节点需要运行而一个节点关闭。对于五个节点，可以关闭两个节点。节点之间的延迟应该在10毫秒左右或更短。

    原始事务数据（调用堆栈，数据库语句，代码级可见性等）不会跨节点复制，而是均匀分布在所有节点上。因此，如果节点发生故障，Dynatrace可以准确估计丢失的数据。这是可能的，因为这些数据通常是短暂的，并且Dynatrace收集的大量原始数据确保每个节点仍然具有足够大的数据集，即使节点在一段时间内不可用。

    如果计划在单独的数据中心中分布节点，则不应在每个数据中心部署两个以上的节点。然后，复制因子为3确保每个数据中心都具有所有度量和事件数据。此外，为了实现无缝连续性，您至少需要三个数据中心，其中一个可能会失败。

    • 硬件

    为防止丢失监视数据，请将每个节点部署在单独的物理磁盘上。要最大限度地降低性能损失，请在具有相同硬件特征的系统上部署节点，如果发生硬件故障，只会影响故障机器上的数据; 没有监视数据丢失，因为所有节点都复制监视数据。性能损失最小化，因为所有节点都在具有均匀分布的工作负载的相同类型的硬件上运行。

    • 处理能力

    构建具有额外容量和可能的节点故障的群集。以100％的处理能力运行的集群没有处理能力来补偿丢失的节点，因此在节点发生故障时容易丢弃数据。计划用于节点故障的部署应该具有比其典型利用率高三分之一的处理能力。

    如果节点发生故障，负载均衡系统的 NGINX 会自动将所有 OneAgent 流量重定向到剩余的工作节点，除了更换故障节点之外，不需要用户操作。
