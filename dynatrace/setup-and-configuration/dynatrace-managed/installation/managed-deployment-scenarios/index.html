    Managed 部署方案

    Dynatrace Managed 的不同可能部署方案详述如下。每个图示描述了各种涉及的组件以及组件之间所需的通信端口。图中箭头的方向表示哪个组件启动通信。

    如果您希望只能在内部访问 Managed 群集，请使用基本设置（下面的方案1）。如果要从外部源启用 DEM 服务和通信，请实现纯 Dynatrace 托管设置（方案2）。如果要将 Dynatrace Managed 集成到现有IT基础架构中，请遵循方案3。

    场景1：基本设置

    安装 Dynatrace Managed 并设置一个或多个集群节点后，您将获得下面所示的部署方案。

    无需进一步配置（从 CMC 菜单中选择 Settings > Public endpoints），群集只能在内部访问，并公开443 REST API，OneAgent 流量和 Web 访问（集群管理控制台和环境 UI）的端口。默认情况下，启用对 Mission Control 的远程访问。这使 Dynatrace 能够对托管群集执行监视和运行状况检查。每个通信信道都使用 TLS 进行保护。

    场景2：纯 Dynatrace 管理设置

    如果希望群集从外部 OneAgent 接收监视数据，或者如果要使用我们的数据体验监视（DEM）服务，则需要将群集公开给外部网络并配置公共 IP 地址。DEM 服务包括：

    • 综合监测
    • 无代理真实用户监控
    • 移动真实用户监控
    • 通过浏览器扩展实现真实用户监控
    • 与 Dynatrace 移动应用程序通信

    处于安全原因，不建议将集群直接暴露给外部网络。因此，我们建议使用一个或多个 Cluster ActiveGates 作为中介代理，以预处理 OneAgent 和 DEM 流量/群集 ActiveGates 将被集群识别，并且可以通过群集管理控制台进行配置，类似于群集节点，下图显示了每种情况的相关通信端口。

    Cluster ActiveGate 需要：

    • 公共 IP 地址
    • 具有有效SSL证书的域名，因为仅使用HTTPS（端口443）以安全方式支持外部通信。此域必须与Web UI域不同。您可以选择自己提供域名和SSL证书，也可以让Dynatrace为您执行此操作。Dynatrace可以代表您生成域和有效的SSL证书。

    对于具有外部主机，应用程序，会话和综合监视的高负载，生产就绪安装，建议您使用相同的域名和证书设置两个负载平衡的 Cluster ActiveGates。对于较小的低负载安装，您只能使用一个 Cluster ActiveGate。此外，您可以考虑为每个环境安装单独的 Environment ActiveGate。

    由于 Environment ActiveGate 在安装时开始与 Cluster ActiveGate 进行通信，因此 Cluster ActiveGate 必须是可操作的，并且必须事先通过公共 IP 地址提供。因此，如果您计划使用 Environment ActiveGate，则部署顺序非常重要：

    1. 设置 Dynatrace Managed 群集
    2. 设置 Cluster ActiveGate 并确保它可以连接到 Dynatrace Managed 群集。此外，请确保 Cluster ActiveGate 具有公共IP地址，并且可从网络外部访问。
    3. 设置您的 Environment ActiveGate 并为外部通信提供 Cluster ActiveGate 公共地址。

    注意：Web UI 流量或群集管理（通过群集管理控制台）应保留在网络内的本地。使用 HTTPS 访问 Web UI。因此需要 TLS 证书。使用自签名证书是可以接受的，但您可能也希望以安全的方式执行 Web UI 流量和群集管理。在这种情况下，您可以提供域名和有效的 SSL 证书，或者这些证书可以由 Dynatrace 自动生成。默认情况下，每个群集都会*.dynatrace-managed.com从Let's Encrypt 获取具有有效证书的子域。另请注意，Cluster ActiveGates 不支持代理 Web UI 流量。
