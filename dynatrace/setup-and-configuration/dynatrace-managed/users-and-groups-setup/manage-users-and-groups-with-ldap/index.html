    通过 LDAP 管理用户和组

    您可以将 Dynatrace Server 连接到外部身份验证服务器，以导入需要访问 Dynatrace Managed 环境的用户组或帐户。通过 LDAP 集成，可以从外部 LDAP 资源访问所有用户。然后，您可以通过访问“ 组” 页面来分配用户组权限和角色，如下所述。

    连接设置

    1. 从 Cluster Management Console 菜单中，选择 User authentication> User repository。

    2. 从列表框中选择外部 LDAP 服务器。

    注意： 切换到 LDAP 身份验证后，本地帐户将停止工作，并且 Dynatrace Server 用户界面将不再提供这些帐户。 但是，无论选择哪种身份验证提供程序，您在安装期间创建的管理员帐户都将继续工作

    1. 输入您的 LDAP 主机地址。如果 LDAP 服务器不使用默认389端口，则可能需要调整端口号。

    2. 指定 LDAP 用户帐户的绑定 DN（可分辨名称），例如，格式为： CN=UserName,OU=OU-name,DC=DomainName,DC=DomainExtension 或任何其他有效的LDAP字符串。

    3. 输入绑定 DN 中指定的 LDAP 用户使用的密码。

    4. （可选）定义额外的连接参数：

    5. 通过启用“ 使用 SSL” 开关启用与 LDAP 服务器的加密通信。

    6. 如果已在 LDAP 服务器上配置了引荐，请设置“ 最大引用跃点数”。

    7. 单击“ 测试连接”以查看 Dynatrace Managed 是否能够访问 LDAP 服务器。在连接测试期间，我们尝试识别您正在使用的 LDAP 服务器的类型。根据此信息，我们将为您提供组和用户查询的默认设置。连接成功后，您就可以查询并导入组和用户。

    群组查询

    成功进行连接测试后，“ 组”查询步骤将变为活动状态。

    1. 在相应的字段中键入查询字符串，以返回要与 Dynatrace 集成的组。

    2. LDAP 目录以树结构组织。组查询的基本 DN 是条目，其中包含组所在的子树。例如，在下图中，有两个包含用户组的子树： OU=Groups,DC=dynatrace,DC=org和OU=Lab,DC=dynatrace,DC=org

      如果要将用户分配给两个子树中的组，则应将组查询的基本 DN 指定为DC=dynatrace,DC=org（父条目）。要仅将用户分配给OU=Lab,DC=dynatrace,DC=org子树的组，请将此子树指定为基本 DN。

    3. 您可以键入 LDAP 筛选器字符串以缩小返回的组的数量。过滤器应包含有关组条目具有的对象类的信息。例如，对于 Active Directory，默认筛选器是：

      (objectClass=group)

      对于 OpenLDAP，默认过滤器是：

      (objectClass=groupOfNames)

      要缩小已使用组的数量，可以使用组名限制扩展过滤器。例如下面的过滤器来缩减由系统的是具有与基团使用的基团group作为objectClass属性和CN与短语开始属性（通用名）PL：

      (&(objectClass=group)(CN=PL_*))

      您可以在此处插入任何其他有效的 LDAP 查询。请记住，LDAP 不区分大小写。

    4. 配置“ 组ID”属性。此属性仅在特定情况下使用。要了解更多信息，请查看下面的匹配用户和组部分。如果不适用，请将此值设置为与组名称属性相同的值。

    5. 配置组名称属性。这是保存组名称的属性，通常称为name（例如，对于 Active Directory）或cn（例如对于 OpenLDAP）。 LDAP 目录中的组名属性值应与“ 用户组”页面上的 LDAP组名称匹配（请参见下图）。请记住，LDAP 不区分大小写。 注意：“ 用户组”页面上的 LDAP 组名称默认设置为组创建期间提供的组名称。

    6. 配置“ 组成员”属性。“ 匹配用户和组”部分详细介绍了此属性。

    7. 单击“ 测试查询”以测试您的设置并验证查询是否有效。

    用户查询

    成功进行连接测试后，“ 用户”查询步骤将变为活动状态。

    1. 在相应的字段中键入查询字符串，以返回要与 Dynatrace 集成的用户。

    2. LDAP 目录以树结构组织。用户查询的基本 DN 是条目，其中包含用户所在的子树。例如，在下图中，有两个子树持有用户：

      OU=Functional,OU=Accounts,DC=dynatrace,DC=org 和OU=Primary,OU=Accounts,DC=dynatrace,DC=org

      要从两个子树中对用户进行身份验证，您应该将用户查询的基本 DN 指定为OU=Accounts,DC=dynatrace,DC=org（父条目）。如果您只想OU=Primary,OU=Accounts,DC=dynatrace,DC=org对子树中的用户进行身份验证，则应将此子树指定为基本 DN。您还可以进一步将系统用户限制为OU=EU,OU=Primary,OU=Accounts,DC=dynatrace,DC=org子树（例如，通过将此子树设置为基本 DN）。

    3. 您可以键入 LDAP 筛选器字符串以缩小返回的用户数。过滤器应包含有关组条目具有的对象类的信息。例如，对于 Active Directory 和 OpenLDAP，默认筛选器是：

      (objectClass=person)

      要缩小经过身份验证的用户数，可以使用任何有效的 LDAP 查询扩展过滤器。例如，下面的过滤器将经过身份验证的用户缩小user为objectClass将属性和department属性设置为指定值之一的用户：

      (&(objectClass=user)(|(department=101)(department=102)(department=103)))

      请记住，LDAP 不区分大小写。

    4. 配置 Login 属性。该属性用于登录系统。

    5. 如果提供的属性不适合您，请调整名字属性，姓氏属性 和电子邮件属性。

    6. 配置组成员资格属性。以下匹配用户和组中详细介绍了此属性。

    7. 单击“ 测试查询”以测试您的设置并验证查询是否有效。

    注意： 测试查询选项（对于组和用户）仅测试基本 DN，过滤器和必需属性的正确性 - 组的组名称属性和用户的登录属性。如果未正确配置非必需属性，则测试查询不会引发错误。此外，他们不会检查用户是否正确分配到组。

    匹配用户和组

    有几种方法可以将用户与 LDAP 目录服务器中的组进行匹配。例如：

    1. LDAP 组条目中的组成员属性（例如，member或uniqueMember）包含用户的DN：

    在这种情况下，不需要配置 Group ID属性。您可以将其配置为与“ 组名”属性相同的值。

    1. 用户条目中的组成员资格属性（例如，memberOf或isMemberOf）包含组的DN：

    在这种情况下，也不需要配置组 ID 属性，因为组的DN用于用户组匹配。您可以将“组 ID”属性配置为与“ 组名”属性相同的值。

    1. 用户条目中的组成员资格属性（例如，gid或group）包含组的 ID。在这种情况下，需要将 Group ID 属性配置为存储引用值的属性。

    在上面的示例中，应将用户查询中的“ 组成员资格”属性配置为gid ，并将组查询中的“ 组 ID”属性配置为gidNumber。

    1. 在下面的示例中，应将用户查询中的“ 组成员资格”属性配置为，group并且应将组查询中的“ 组 ID”属性配置为cn（与本例中的“ 组名”属性相同）。

    ``` dn: cn=user 3,ou=People,dc=example,dc=com objectClass: person cn: user 3 group: test

    dn: cn=test,ou=Groups,dc=example,dc=com objectClass: groupOfUniqueNames cn: test ```

    将 Dynatrace Managed 组映射到 LDAP 组

    有关 Dynatrace Managed 中可用的用户组权限的信息，请参阅哪些角色和用户组可用？

    从 LDAP 成功配置组和用户后，需要将监视环境角色分配给用户目录中的组。默认情况下，不会向导入的组授予监视环境权限。

    注意： 在执行此步骤之前，用户将无法访问监视环境。

    1. 从导航菜单中，选择用户身份验证>用户组。
    2. 从 LDAP 导入的组列表中，选择要配置的组名称。
    3. 您可以通过为此组授予全局管理员权限来为任何特定组分配集群管理员权限。然后，该组中的所有用户帐户都将拥有管理员权限。
    4. 在“ 权限”部分中，手动设置每个环境的权限。

    注意： Dynatrace Managed 显示的用户列表仅显示属于已分配 Dynatrace 管理角色的组成员的帐户。
