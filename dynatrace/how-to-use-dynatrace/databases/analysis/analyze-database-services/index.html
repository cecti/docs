    分析数据库服务

    Dynatrace 监视应用程序执行的每个数据库语句的响应时间，故障率，吞吐量等。

    分析数据库的性能

    1. 从导航菜单中选择数据库。
    2. 选择要分析的数据库。然后显示数据库服务的概述页面（参见下面的示例）。

    数据库服务概述页面上的信息图提供了对数据库性能各个方面的深入了解，包括 SQL 查询或过程，SQL 修改，SQL 事务，任何检测到的问题或可用性问题，热点等。包含的磁贴的确切列表取决于您正在监控的数据库技术。每个磁贴都包含响应时间，故障率和吞吐量指标。

    要分析导致这些类型语句的单个语句，单击相关磁贴上的“查看数据库语句”按钮。

    声明限制

    默认情况下，数据库语句的长度限制为 4KB（KiloBytes）。此设置不可配置。超出此限制的数据语句将调整为 4KB。

    在所选时间范围内执行的所有数据库语句都列在“ 详细信息”页面底部的“ 数据库语句”部分中（请参阅下面的示例）。默认情况下，数据库语句基于中值响应时间进行排序。您可以选择其他排序选项 - 只需从排序依据下拉列表中选择一个。您还可以通过单击页面顶部的“ 添加过滤器”按钮来过滤语句列表。

    单击任何语句以查看该语句的完整详细信息。对于每个语句，Dynatrace 显示执行次数 /分钟数，总执行次数，总时间，响应时间中位数，响应时间最慢10％的执行次数和失败率。

    数据库服务回溯

    单击“ 分析回溯”按钮以访问所选语句的回溯分析。该回溯页面提供了触发 SQL 语句的选定排列的点击详细信息，还允许您查看哪个服务链和哪个代码指向特定的语句。

    通过在回溯中选择特定的应用程序，您可以从这个应用程序中找到触发您正在分析的数据库语句的所有用户操作。在上面的例子中，您可以看到约 43k 数据库请求仅来自于 5.32k 左右的用户操作，并且正好有3个不同的用户操作触发了该语句。您还可以看到触发此语句的服务链。通过单击该语句并进一步研究，我们可以看到约 5.3k 用户操作导致 Apache web 服务器上的近 5.6k 请求，接近1:1的比例。然而，一旦这些请求到达我们的 easyTravel 客户前端服务，这些5.6k的请求将导致对 travelyservice 的约2万次调用。需要更多的调查来确定原因。为此，在服务调用链中选择easyTravel Customer Frontend。

    服务回溯的下半部适应新的选择，您现在看到所有这些用户操作都调用 easyTravel 客户前端的/CalculateRecomendations请求。如果其中一些请求失败了，您可以在相应的选项卡中分析它们。这里我们更感兴趣的是导致数据库调用的代码，因此我们选择 Stacktrace 选项卡来查看上下文中执行的代码。

    响应时间分布

    要了解该命令的响应时间分布，请单击 Analyze Outliers 按钮。响应时间分布页面显示在不同响应时间范围内的请求数量。选择图表中的任意一列来分析属于该特定响应时间范围内的特定请求。这使您能够查看是否有经常调用的语句，并且它们对响应时间的贡献实际很长。

    分析性能下降

    在分析性能下降时，Dynatrace 数据库监控的真正威力显现出来。

    您可以立即查看哪些语句对检测到的减速负责 - 它们在页面底部的语句列表中突出显示。在下面的示例中，检测到的响应时间降级是由两个语句（Aggregations in BookingCollection和Aggregations in JourneyCollection）引起的。

    要查看哪个服务执行了导致数据库速度减慢的特定语句，请单击该语句的 Service backtrace 按钮（请参阅Aggregations in BookingCollection下面的示例）。

    在本例中，它显示了执行导致速度下降的语句的服务被称为easyTravel-Business。

    分析数据库错误

    数据库服务概述页面上的每个磁贴都具有故障率图表。

    单击“ 故障率”图以打开“ 详细信息”页面的“ 故障”选项卡。

    页面底部的失败数据库语句表会立即显示哪些语句失败以及各自的失败率。

    单击“ 查看故障详细信息”按钮以了解故障的根本原因。

    Dynatrace 已确定此示例中高故障率的原因是 MySQL 语法错误异常。这种类型的详细错误信息可以为您的开发人员提供解决数据库问题和消除未来错误所需的信息。

    具有最高故障率的语句列在数据库服务概述页面的“ 当前热点”部分中。单击该语句以查看该语句的“ 详细信息”页面的“ 失败”选项卡。

    SQL 服务流程分析

    服务流可视化由环境中的每个服务请求触发的服务调用序列。使用服务流，您可以从单个服务，请求或其过滤子集的角度查看服务调用流，包括数据库服务调用。

    查看特定数据库服务触发的服务流

    1. 从导航菜单中选择“ 交易和服务 ”。
    2. 单击要分析的服务以打开服务的概述页面。
    3. 在“ 了解依赖关系”部分中，单击“ 查看服务流”按钮。

    虽然 SQL 分析可以在服务分析工作流的多个位置使用，但它是服务流视图中数据库的主要分析视图。

    下面的示例显示了一个数据库服务流，该流已经过筛选，以专注于特定的调用链。您可以看到，每当 Customer Frontend 服务调用 JourneyService 时，25%的响应时间贡献可以归因于 EasyTravel-Business MongoDB 服务。

    要查看分析时间框架内所选进程执行的数据库语句，请单击“查看数据库语句”按钮。这非常有价值，因为它可以帮助您理解为什么数据库要花费一定的时间。

    要显示哪些语句具有最高的总响应时间，请从“ 排序”下拉列表中选择“ 按总时间排序”。

    要了解响应时间的范围，请单击“ 分析异常值”按钮。

    要了解随时间经历的每个单独 SQL 语句的演变以及返回的平均行数，请单击“ 详细信息”。

    单击 Service backtrace 按钮以查看哪些用户单击导致此 MongoDB 语句的最慢执行。

    如您所见，Dynatrace 数据库语句分析视图提供了功能强大的数据库使用情况分析。
