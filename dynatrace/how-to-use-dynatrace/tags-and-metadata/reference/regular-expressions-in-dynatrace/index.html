    Dynatrace 中的正则表达式

    如果希望在字符串中查找或提取特定模式，正则表达式非常强大。然而，权力越大，责任越大。正则表达式可能导致意外的结果（惰性匹配和贪婪匹配），并且它们可能很快成为资源密集型的（灾难性回溯和重复捕获组）。

    正则表达式也经常被误解。如果使用不当，小正则表达式很容易消耗大量处理能力。例如，考虑这个正则表达式(.*)+b。此表达式可能看起来不危险，但如果针对不包含的较大字符串执行b该表达式，则此表达式将导致循环持续超过10秒。

    由于这些原因，我们在 Dynatrace 中限制了正则表达式。然而，在几乎所有情况下，这都不会限制您的功能。不过，这可能会迫使您学习更多关于正则表达式的知识。

    限制条件

    阅读下面提供的限制条件，以了解如何在 Dynatrace 的上下文中使用正则表达式。

    不允许量化或重复组

    通常不需要量化组，因为它们很容易失控，我们不允许它们。要更好地理解这一点，请访问 https://regex101.com 并评估以下正则表达式：

    Regex: (a+)+b
Value:aaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaa

    您会注意到它将超时。在本例中，问题在于捕获组的回溯特性。我们决定不允许完全使用重复组。

    捕获组中的贪婪匹配与占有匹配

    Dynatrace 不允许捕获组内的贪婪匹配。请改用懒惰或占有匹配。

    正则表达式的原型用法就是这个术语.*。贪婪的匹配被定义为“在零和无限次之间匹配，尽可能多次，根据需要回馈（贪婪）”。这里的问题是“多次”和“回馈”。人们通常使用它来匹配某些东西，直到满足边界条件。考虑以下正则表达式匹配：

    Regex: key: (.*);
Value: something. key: value; something else

    正则表达式引擎将首先搜索key:然后贪婪匹配该组，(.*)直到字符串结束。随后它检查是否有一个; 后相匹配; 这将永远不会是这样。然后它必须一次回溯一个字符。每次它首先检查该组是否仍然匹配，直到找到;。作为一个小细节，它还必须每次记住该组的新值; 因为它是一个捕获组。所有这些都非常昂贵且效率低下; 更重要的是，随着要匹配的字符串的长度增加了复杂性。

    更好的方法是：

    Regex: key: ([^;]*+)
Value: something. key: value; something else

    只要;之后没有找到，此匹配将匹配key:。加号还将其转换为占有性匹配，定义为：“零和无限次之间的匹配，尽可能多次，不回馈（占有）”。这里的关键点是它没有回溯。换句话说，一旦某些东西匹配，那么引擎将不会重新考虑并且不会回头但只会前进。这也是为什么我们需要从一切（.）到一切都改进我们的匹配，但不是;。我们需要告诉它匹配一切，但不是;。

    如果你在正则表达式调试器中比较第一个正则表达式与第二个正则表达式（遵循这两个链接），你会发现第一个正则表达式需要42个步骤而第二个步骤需要9步！这就是为什么我们建议你一般应该使用占有量词。

    不允许捕获组进行简单匹配

    正则表达式中的捕获组用于提取信息。有时这个概念被误用于简单的查找匹配。由于这只会增加开销并且不需要，因此 Dynatrace 不允许这样做。如果您仍需要一个组，请使用原子组代替。

    Dynatrace 只允许在需要提取值的地方使用一个捕获组。这里的逻辑是，如果正则表达式包含一个组，Dynatrace 将使用它。否则，将使用完整匹配进行提取。

    这意味着您根本不需要使用捕获组。在大多数提取方案中，只需定义您想要获得的内容即可。在提取案例中，以下正则表达式将捕获所需的值 - 无捕获组。

    如果要从用户代理字符串中捕获 Windows 版本，可以使用以下正则表达式：

    Regex: Windows NT[^)]+
Value: Mozilla/5.0 (Windows NT 10.0) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/56.0 Safari/537.36

    这将捕获Windows NT 10.0并且不需要捕获组。

    Dynatrace 不允许反向引用

    反向引用通常用于解决分层匹配或递归匹配，这两种匹配都很可能代价很高，并且在 Dynatrace 中不需要。

    其他常见的误解

    通常人们认为他们需要匹配完整的输入文本。这通常导致像这样的正则表达式：

    Regex: .*key: [^;]*+.*
Value: something. key: value; something else

    这将匹配整个字符串，而不是只匹配所需的内容。这是基于这样一种误解，即给定的正则表达式总是匹配全文(通常用于查找子字符串)。在 Dynatrace 中，正则表达式条件通常搜索模式，并不强制匹配整个输入。不需要前后匹配(.*)，所以请不要使用它们。
