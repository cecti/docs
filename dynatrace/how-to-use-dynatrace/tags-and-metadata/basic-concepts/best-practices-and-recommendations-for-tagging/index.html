    标签的最佳实践和建议

    标记是一种强大的机制。然而，为了获得它的好处，应该谨慎并以有意义的方式使用标签。为了指导您实现这一目标，我们向您提供了具体的建议和最佳实践，如下所述。通过基于元数据的自动标记，可以自动生成标记，并使用 Dynatrace 自动检测到的特定元数据值将标记分配给受监视的实体。

    强烈建议您在设置自定义图表，过滤服务或主机列表以及警报配置文件时使用标签而不是特定实体名称。基本上，在您可能想要引用一组实体的任何地方，您应该使用标记而不是定义直接引用。这可确保更轻松地维护系统配置和集成。即使在频繁添加和删除新组件的高度动态环境中，您也可以使用此方法确保实体过滤器（例如，自定义过滤的仪表板磁贴）即使在监视环境中添加和删除实体时也保持最新状态。标记还可以更轻松地将自定义配置从一个环境传输到另一个环境。

    无论何时创建仪表板，图表或警报配置文件，都应考虑定义标记以引用相关实体。这并不意味着您应该为所有事先定义通用标记。标签应该是基于用例的。但是，您应该为您认为可能变得重要的任何内容添加或定义元数据作为标记的基础。

    主机和元数据标签

    建议在主机的已部署系统上定义额外的元数据。您应该尽可能多地发送额外的元数据。Dynatrace 可以使用这些额外的数据来定义标记、管理区域，或者在图表、仪表板等内部动态地定义。还建议对组织的一些元数据进行标准化。您可以在部署过程中以自动方式将它们保存到相应的配置文件中。

    标准化的重要方面是：

    • 所有者/团队/业务部门/业务部门
    • 环境：筹划/生产
    • 版本（如适用）
    • 重要性/严重性（与警报配置文件相关）

    对于主机标签，建议您不要在已部署的系统上定义标签。原因是这很麻烦，需要大量的预先计划。它也很难改变。只有在您想要标记的内容很少更改并且您需要全局创建过滤器时才执行此操作（可能的示例：重要性/严重性）。另一种情况是，如果您的工作方式要求您在部署期间以自动方式定义标记。

    应用程序元数据和标签

    对主机推荐的内容对应用程序和进程也有效。通常，您应该考虑其他元数据和标准元数据，而不是标记。

    对于 WebSphere 或 Tomcat 等应用程序，您应该使用环境变量DT_CUSTOM_PROP来定义元数据。此变量需要在应用程序启动时出现。对于 WebSphere，您可以在 JVM 部分的 WebSphere 控制台中执行此操作。对于 Tomcat 和其他人，只需将其定义为启动脚本的一部分。

    使用 Kubernetes，使用Kubernetes注释来定义元数据。Dynatrace 将自动获取所有注释。AWS 也是如此。Dynatrace 还将获取 AWS 标记，但在这种情况下，标记不是元数据。当然，您仍然可以使用它们来定义其他标记或将其用于任何其他目的。

    要为 Cloud Foundry 定义 Dynatrace 标记，您可以利用一个或多个名称dynatrace为子字符串的 Cloud Foundry 服务实例。

    自动标签

    建议您根据现有或自定义元数据配置自动标记规则，以定义图表，警报等过滤器集。这些标签和规则可以随时更改和调整，几乎可以立即应用，而无需对受监控的环境或应用程序进行任何更改。

    CMDB，API 和自动化

    如果您有要定义和维护标记的 CMDB，则可以这样做并使用API将这些标记与 Dynatrace 同步。这意味着您可以定义标签并通过 API 将它们分配给实体。这应该用于已经在 Dynatrace 之外的位置维护的常用标记。

    管理区

    管理区域是 Dynatrace 中的一个重要概念，在大型环境中非常重要，因为它们可以让大型组织正常工作。管理区域基于与自动标记相同的思想，但设计用于明确定义属于同一实体组或具有共同安全级别的实体组。

    建议为所有团队定义区域，这些区域指定他们应该查看的内容和不同的业务线。为不同的支持团队添加不同的区域也很有意义，这样就可以使用大量的数据，并引入安全层。每个用户都可以访问多个区域。区域可以重叠，因此您不必担心将一个实体分配给特定的区域。因此，共享实体可以位于许多或所有区域。

    创建管理区域的最佳方法是根据实体元数据（包括自定义元数据）定义规则。但是，如果要在 Dynatrace 之外维护管理区域，还可以将它们基于标记。

    问题，管理区域和标签

    问题也可以通过标签过滤。这基本上意味着可以专注于与具有特定标签的实体相关的问题。这也是通知集成可以利用标签的方式。可以定义为仅将涉及具有特定标签的实体的问题发送到特定通知集成。

    命名规则和命名约定

    Dynatrace 自动提供名称，但是它们不能让您快速识别应用程序或服务属于何处。为此，建议您使用服务命名规则和流程组命名规则。这可以在 Dynatrace 中使用从监控应用程序导入的元数据来完成。

    强烈建议您使用与业务单位（BU或LOB）或其他所有者指定相关的名称为流程组和应用程序添加前缀。这种命名标准将自动应用于新的和现有的实体，并可根据新出现的需求在未来加强。这也是为什么定义每个应用程序可以提供的最小元数据集的原因之一。
