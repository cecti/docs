    用户操作

    用户操作是与web浏览器的交互，包括对web服务器的调用，web服务器可能有多个嵌套调用。

    用户操作类型

    用户操作可以是页面加载、XHR操作或自定义操作。这些操作类型之间的关键区别在于计算操作持续时间的方法，对于每种类型，都有不同的可用度量标准。

    页面加载

    页面加载是浏览器中的实际页面加载。如果您在浏览器中输入 URL 并按 Enter 键，则会发生页面加载。在页面加载期间，加载了许多资源，包括图像，HTML 和 CSS。

    在这种情况下，操作持续时间是完整页面加载所需的时间。更具体地说，用户动作的开始navigationStart时间以 W3C 时间开始（如果可用）。如果不可用，则在浏览器中初始化 Real User Monitoring JavaScript 代码时开始时间。结束时间是最后一个onload处理程序完成其任务的时间。该onload处理器是在 JavaScript 事件处理程序这是我们用来调用 JavaScript 的执行的页面，帧或图像后已经完全加载。如果处理程序XMLHttpRequests启动了任何操作（请参阅下面的XHR操作）onload，则用户操作在XMLHttpRequest完成时结束。

    资源计时步骤

    以下度量用于绘制页面加载过程中特定步骤的持续时间。

    测量 描述 根据W3C规范定义
    DNS 花费或解析域名的时间。 window.performance.domainLookupEnd - window.performance.domainLookupStart
    Connect 建立从浏览器到 Web 服务器的套接字连接所花费的时间。 window.performance.connectEnd - window.performance.connectStart
    SSL 建立从浏览器到Web服务器的安全套接字连接所花费的时间。 window.performance.connectEnd - window.performance.``secureConnectionStart
    URL Redirection HTTP 重定向后花费的时间。 window.performance.redirectEnd - window.performance.redirectStart
    Request 等待文档响应的第一个字节所花费的时间。 window.performance.responseStart - window.performance.requestStart
    Response 下载文档响应所花费的时间。 window.performance.responseEnd - window.performance.responseStart
    Total 传递响应和OnLoad事件之间的时间。 window.performance.loadEventEnd - window.performance.domLoading

    XHR action

    大多数现代应用程序，包括单页应用程序，都依赖于下载框架并初始化页面的单页加载。然后，通过 JavaScript 更改页面的 DOM，并通过XmlHttpRequest或fetch()与 web 服务器进行所有通信。

    Dynatrace 不断跟踪用户与每个页面的交互。如果用户交互导致xmlhttprequest或fetch()调用，则创建一个 XHR 操作。Dynatrace 还检测在初始XHR的回调中是否触发了其他 XHR，等等。在本例中，Dynatrace 将等待所有请求完成。通过监视 DOM, Dynatrace 还可以识别回调中添加的资源。然后，Dynatrace 会等到这些资源下载完成后才停止操作。

    XHR 操作从用户单击网页上的控件开始。所有指标都是根据此时间点计算的，并且基于触发用户操作的初始 XHR。

    获取 API

    Fetch API 提供了一个获取资源(包括通过网络)的接口。它类似于XMLHttpRequest,但 API 提供了一个更灵活的特性集。通用的定义Request,Response和其他网络请求对象获取允许他们使用在需要的任何时候,无论是服务工人,缓存API,或者任何处理或修改请求和响应。Fetch 还支持跨源资源共享(Cross Origin Resource Sharing, CORS)。

    基于 Fetch API 的用户操作在 Dynatrace 中显示为 XHR 操作。您可以将 Dynatrace 配置为自动检测和捕获 Fetch API 请求信息。

    自定义用户操作

    与其依赖默认的用户操作生成，您可能希望通过将其他用户操作直接添加到应用程序的 HTML 中来微调实际的用户监视。如果我们的自动用户操作生成没有捕获特定的操作，或者您希望在应用程序监视中引入特定的细粒度计时，那么这将非常有用。例如，您可以测量打开一个 JavaScript 专用下拉菜单需要多长时间，或者测量一些 JavaScript 代码的持续时间。要定义自定义操作，可以使用 JavaScript API 进行实际的用户监视。

    用户操作命名规则

    许多应用程序允许用户通过不同的 UI 控件来实现相同的目标。在监视此类应用程序时，很难区分具有相同结果和目标，但使用应用程序UI的不同部分执行的操作。同样，如果将应用程序的UI翻译成多种语言，那么相同的应用程序功能或控件可能会以不同的名称出现。使用用户操作命名规则，Dynatrace 可以检测到这些细微的变化，并智能地对相关的用户操作(例如，(例如，实现相同目标的用户操作)分成逻辑组进行监视。

    Dynatrace 会自动sessionid从用户操作名称中删除某些常用令牌（例如，jsessionid对于 Java 容器，sessionidPHP 的默认值CFID以及CFTOKENColdFusion 的默认值）。尽管如此，您的环境中可能存在许多会话 ID 变体。如果 Dynatrace 不会自动识别并从您遇到的某些用户操作名称中删除会话 ID，则需要为这些用户操作配置自定义命名规则。

    关键用户操作

    大多数 web 应用程序都包含一些对数字业务的成功特别重要的用户操作(例如，注册操作、结帐或产品搜索)。这类“关键”用户操作的执行时间可能比其他用户操作要长，因此需要特殊处理(或者可能需要比平均持续时间短)。

    例如，您已经将全局 Apdex 阈值设置为3秒。虽然这个阈值对于大多数用户操作来说是可以接受的，但是对于注册用户操作来说，它可能是不可接受的，因为注册用户操作的持续时间必须更短。相反，可能您有一个复杂的搜索操作，需要更长的阈值5秒。

    关键用户操作特性允许您为此类操作定制 Apdex 阈值。它还允许您使用专用的仪表板平铺来监视关键操作，并跟踪历史趋势。

    注意：

    Dynatrace 允许您为每个应用程序创建最多20个关键用户操作。如果您的申请需要更高的限制，请联系您的 Dynatrace One 团队或您的销售代表。
