    通过“My web application”占位符定义您的应用程序

    OneAgent 安装在主机上后，它会监视主机上运行的所有应用程序。所有监视数据都封装在一个名为“My web application”的示例占位符应用程序中。我们提供此占位符应用程序的原因是为了提供更大的灵活性，因为我们可以为您提供应用程序的组织方式。例如，有些情况下您可能希望将多个域合并到一个应用程序中。对于以多种语言提供的站点，通常就是这种情况，例如www.myshop.de和www.myshop.at。

    通过“My web application”，您将找到一个名为 Top domains 的部分（请参阅下面的完整说明）。这可能是您定义自己的应用程序映射的起点。您可以通过打开顶级域部分的完整详细信息，轻松地将识别的域映射到单独的各个应用程序中。如果要创建更多应用程序，更改现有应用程序映射，或者如果需要定义更复杂的规则，查看URL而不仅仅是域，则可以使用“ 应用程序检测和RUM”设置页面（可在“设置” >“ Web和移动监视”中找到）。请注意，这些设置仅适用于Web应用程序。

    以自动方式定义应用程序

    以自动方式定义应用程序

    1. 从导航菜单中选择应用程序。
    2. 单击 My web application 占位符应用程序。
    3. 向下滚动以查找前3个包含的域面板。此面板包括包含由环境中的 OneAgent 自动检测到的最大数量的操作的域。
    4. 点击查看全部详细资料。
    5. 从顶级域下显示的列表中选择一个域，然后单击“ 传输域”下的箭头将其展开。
    6. 单击创建新应用程序。您的应用程序将在“ 应用程序”页面上创建并列出。从现在开始，在此域上监视的所有用户操作都将映射到新创建的应用程序。或者，您可能希望将域添加到现有应用程序，以防某些应用程序已创建。为此，请展开列表框，选择一个应用程序，然后单击“ 传输”。

    由于您可能希望为应用程序使用更直观的名称，因此可以轻松地重命名它。要重命名应用程序：

    1. 从导航菜单中选择应用程序。
    2. 单击新创建的应用程序以访问应用程序的概述页面。
    3. 单击“浏览”按钮（...），然后选择“ 编辑”。
    4. 在页面顶部显示的文本框中键入您喜欢的名称。请注意，应用程序名称必须是唯一的。

    使用应用程序检测规则定义应用程序

    使用应用程序检测规则定义应用程序

    1. 从导航菜单中选择“设置”，然后单击“ Web和移动监视”。
    2. 选择应用程序检测和 RUM。在应用程序检测规则下，您可以查看已定义规则的列表。对于以自动方式定义的每个应用程序，将自动生成检测规则并将其添加到列表的末尾。规则按顺序应用，列表顶部的规则优先于较低规则。
    3. 点击创建应用程序的检测规则。使用此页面上提供的选项为您的应用程序创建适当的检测规则。

    最佳做法和建议

    • 根据团队所有权定义您的应用程序，以便您可以轻松利用管理区域进行访问限制。
    • 基于所使用的技术堆栈定义应用程序可能是有意义的，以便应用正确的设置并使特定设置的管理更容易。例如，通常只需要对大型应用程序的特定部分和特定用户操作命名规则激活对特定 XHR 框架的支持。由于使用的技术不同或者这些部件的所有者不同，将这样大的应用程序分成较小的应用程序可能会有所帮助。

    • 您应该避免将“我的Web应用程序”重命名为特定的应用程序名称，因为“My web application”将包含所有未包含在任何现有应用程序规则中的域上的所有用户操作。如果重命名“My web application”，则可能难以将其与其他应用程序区分开来。

    • 通过顶级域分隔应用程序始终是一个好主意，因为 Dynatrace 无法将顶级域中的用户操作与用户会话相关联。这种关联是通过 cookie 完成的，因此只有在可以在同一顶级域上设置 cookie 时才有效。例如，www.dynatrace.com 和 blog.dynatrace.com 的用户操作可以在一个应用程序中捕获，因为 cookie 可以设置为 dynatrace.com，但是 www.dynatrace.com 和 www.internal-dynatrace.com 的流量无法在一个用户会话中捕获。您仍然可以根据域分离用户操作，但用户会话永远不会有来自两个域的操作。

    • 应将低流量应用程序组合在一起。如果基于每分钟少于10个操作的域创建应用程序，Dynatrace 将不会自动检测此新创建的应用程序的异常。Dynatrace 依赖于稳定的应用程序流量来正确学习多维基线并自动报告应用程序问题。您可以在应用程序的异常检测设置中更改当前的异常检测设置。虽然这个建议与上面提到的建议相矛盾，但对于低流量应用，将它们组合起来可能是有意义的。

    • 对每个请求按顺序处理应用程序规则。更多规则意味着更多的处理时间，并且在 OneAgent 中处理规则时，您应该尝试将列表顶部流量最多的最常用应用程序的规则。只要一个规则匹配，就不会再处理其他请求。

    • 应首先定义更具体的应用程序检测规则，而更通用的规则应位于列表的底部。假设您要创建一个名为“A”的应用程序，将在其上映射以下两个域：

    http://www.mydomain.com/cms/directory/shop/index.html` `http://shop.differentdomain.de/index.html

    和以下域名的另一个应用程序“B”：

    http://newdomain.co.uk/hello/shop.html

    如果基于该shop值创建通用分组规则，则所有三个域将分组到同一Web应用程序中以进行监视。因此，您应首先定义更具体的规则，例如“如果 URL 以shop.html... 结尾”，以便只将第三个URL映射到应用程序“B”。然后，您可以根据shop值安全地定义通用规则，因为第三个 URL 已经映射到先前的应用程序，因此不会包含在应用程序“A”中。
