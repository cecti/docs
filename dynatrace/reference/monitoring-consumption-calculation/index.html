    监控消耗计算

    Dynatrace 监控消耗基于监控应用程序和相关服务期间 Dynatrace 环境使用的各种类型的监控单元。下面概述了这些监测单元类型的详细信息以及这些单元的消耗方式。

    除非另有说明，否则此处说明的监控单元消耗详细信息适用于 SaaS 和托管部署。要开始使用 Dynatrace，请联系点击这里 。您的销售代表将为您提供更多详细信息。

    本页仅供参考。Dynatrace 免费试用版和/或您的 Dynatrace 许可证的条款将适用于 Dynatrace 产品或服务的任何使用。

    应用和基础设施监控

    对于通过 Dynatrace OneAgent 提供的应用程序和基础架构监视，监视消耗基于主机单元（Host units）。

    主机单位（Host units）

    Host units 通过安装 OneAgent 的系统上可用的内存（RAM）量来衡量。1 host unit 相当于 16 GB 的 RAM（或其任何部分）。

    最大内存 主机单元（全栈*） 主机单元（云基础架构**）
    1.6 GB 0.10 0.03
    4GB 0.25 0.075
    8 GB 0.50 0.15
    16 GB 1 0.3
    32 GB 2 0.6
    48 GB 3 0.9
    64 GB 4 1.0
    80 GB 5 1.0
    96 GB 6 1.0
    112 GB 7 1.0
    nx16 GB n 1.0

    *当主机上的 RAM 量介于上表中列出的值之间时，该数字将向上舍入。例如，具有 12 GB RAM 的主机消耗 1 个主机单元，因为它被视为 16 GB 主机。

    **对于云基础架构监控模式，适用相同的舍入原则，但主机使用的 host units 数量上限为 1。

    全栈 vs. 仅应用

    使用 Universal Injection 集成 OneAgent 时，OneAgent 的操作基于应用程序级别（即仅应用程序监视）。在这种情况下，如果主机级别（物理或虚拟）上未安装 OneAgent，则 host unit 计算的内存检测是在应用程序运行的每个操作系统实例（例如，容器，Azure AppService 实例，Solaris 区域）上完成的，同时会考虑在此图层上设置的内存限制（例如，容器内存限制）。

    如果未设置内存限制，则可能导致更高的 host units 需求，因为报告的可用内存通常是底层主机内存。

    例

    具有 16 GB RAM 的 Docker 主机，运行 2 个容器，仅使用应用程序进行监控：

    变化-1：如果没有设置内存限制，则总共将消耗 2 host units，因为每个容器报告 16 GB RAM。

    变化-2：每个容器的内存限制设置为 4 GB，总共将消耗 0.5 host units，因为每个容器报告 4 GB RAM。 在全堆栈监视模式下监视 Docker 主机将消耗 1 host unit，因为从 Docker 主机报告的内存用于许可（1x 16 GB RAM）。 总 host-unit 消耗量是根据所有模式和受监控系统的所有 host units 的总和计算的。

    IBM z/OS 上的仅应用监控

    在IBM z/OS（CICS 和 IMS）上运行的代码模块基于 MSU 单独许可，不会导致 host units 或 host unit hours 的消耗。

    许可证 MSU 的数量需要与 Dynatrace 检测的 CICS 或 IMS 区域的累积 MSU 容量相匹配。Dynatrace 不提供 MSU 消耗/指标。因此，需要通过 SMF 记录手动评估消耗。

    仅应用程序监控（PaaS）

    仅应用程序（PaaS）监控消耗（可用于 Kubernetes 和 OpenShift）基于标准 host-unit 消耗计算。

    主机单位小时（Host unit hours）

    Host unit hours 表示一段时间内 host unit 的消耗。1 host unit hours 相当于 1 host unit 消耗 1 小时。具有 16 GB RAM（即 1 host unit）运行一整天的主机消耗 24 host unit hours。

    主机单位小时计算示例

    例如，假设您有 1,000 host unit hours 可用，并且您希望监视具有 64 GB RAM（相当于 4 host units）的主机。如果您让主机运行一整天，它将消耗 96 host unit hours。

    4 (host units) x 24 (hours a day) = 96 (host unit hours)

    1,000 host unit hours 将在超过 10 天内消耗。

    4 (host units) x 24 (hours) x 10 (days) = 960 host unit hours

    免费试用 host unit hours

    Host unit hours 用于 Dynatrace 免费试用。当您申请 Dynatrace 免费试用时，您将获得一定数量的 host unit hours 来评估 Dynatrace。

    将 host unit hours 应用于需求高峰并选择项目

    （仅限 Dynatrace SaaS）如果您事先知道由于假期需求或短期项目（例如，在黑色星期五或一次性测试计划中）将超出您的主机单位基本配额，您可以使用 host unit hours 而不是 host units 来管理可变流量峰值。例如，如果您在黑色星期五有一个 9,000 host unit hours 和 100 host units 的池，则您需要更多主机来扩展您网站上增加的流量。在这种情况下，您可以选择在一天内使用所有 9,000 host unit hours。这样您就可以将另外 375 个主机（最多总共 475 个）连接到 Dynatrace 一天。

    9,000 (host unit hours) / 24 (hours) + 100 (base quota of host units) = 475 (max. host units)

    Host 超额

    如果您已配置分配 host units 以监控主机并且您有权超过此数量（即，您的帐户允许超额），则超出部分将以 host unit hours 计算。例如，如果您已安排监控多达 10 个 host unit（最多 160 GB 的总 RAM）并且您的帐户允许超额使用，如果您连接另一个等同于 2 host units 的主机，则您将拥有 12 host units，因此，总数将超过您的配额 2 host units。如果您使用 12 host units 继续监控主机一整周，您将累计超过 336 个 host units。

    2 (host units) x 24 (hours a day) x 7 (days) = 336 (overage host unit hours)

    超额和多种环境

    如果您的帐户有多个监控环境，例如，一个用于开发，另一个用于生产，那么超额计算是按帐户计算的，而不是按环境计算的。只有在超过帐户配额时，才会产生超额。

    例如，您获得了 100 host units的许可，并且您有两个环境，一个用于生产，另一个用于测试。您将 80 host units 分配给生产环境，将 20 host units 分配给测试环境。您的许可授权您进行超额（您可以在 host units 圈下方的帐户概述中看到此信息）。如果生产使用 70 host units 但测试使用 30 host units，则不会超过 100 host units 的总帐户配额，因此不会产生超额。仅当两个环境都使用超过 100 host units 时才会产生超额。

    Host units 的并行

    Dynatrace 用真实并行情况计算 host units。并行运行的所有主机用于计算 host units 以检查 host units 配额。每分钟检查 host units 配额。例如，您有一个 16 GB（相当于 1 host unit）的主机，运行时间为 10：00-10：30。在10:30，您将使用 1 host unit 启动另一台主机。由于主机不并行运行，Dynatrace 将计为 1 host unit。如果您在 10:00 启动第一台主机并在 10:30 启动另一台主机并在 11:00 关闭，则 Dynatrace 计为 2 host units，因为它们同时并行运行。

    数字体验监控

    数字体验监控（DEM）单元用于使用 Dynatrace 真实用户监控，模拟监控和会话重放。下表说明了每种能力类型和计量单位消耗 DEM 单位的比率。

    测量单位 能力 每单位计量的消耗量
    模拟行动 浏览器监视器，浏览器单击路径 1.0 DEM
    模拟要求 HTTP监视器 0.1 DEM
    用户会话* 真实用户监控（禁用会话重播） 0.25 DEM
    用户会话* 真实用户监控（启用会话重播） 1.0 DEM

    * 只有真实用户的用户会话计入您的用户会话消耗中。计算监控消耗时，不计算模拟用户和“机器人”的用户会话。

    真实用户监控（RUM）

    单个真实用户监控会话（即“user session”）被定义为用户与基于浏览器的Web应用程序或本地 iOS 或 Android 移动应用程序之间的间隔内至少两个用户操作之间的交互序列。用户操作是触发 Web 请求的按钮单击或应用程序启动（例如，页面加载或页面视图导航）。仅包含一个用户操作的交互被视为“退回”并且不计算在内。同时与多个 Web 应用程序或应用程序交互的用户为每个 Web 应用程序或应用程序消耗一个会话，除非交互被视为“退回”。会话结束时，a）运行 Web 应用程序的浏览器已关闭或已处于非活动状态超过 30 分钟，b）移动应用程序已关闭或客户端处于非活动状态超过 30 分钟，或 c）连续 60 分钟后与 Web 应用程序或应用程序的交互。

    如果您设置了年度 RUM 会话配额，则您的使用情况将每年重置一次。

    模拟动作和请求

    一个 浏览器监视器（browser monitor） 或 浏览器点击路径（browser clickpath） “模拟动作”是与触发，其包括触发 XHR 或获取请求的页面加载，导航事件或动作的 web 请求的模拟浏览器的交互。浏览器监控器执行单个模拟交互（例如，测量单个 URL 的性能和可用性），并在每次执行时使用一个模拟操作。Clickpath 监视器是预先记录的模拟动作的序列。Clickpaths 每次触发 Web 请求的交互都会占用一个动作（actions）。不触发 Web 请求的向下滚动、按键或单击不计为动作（actions）。

    一个 HTTP 监视器由一个或多个 HTTP（S）的请求（例如，GET，POST，HEAD 请求）。HTTP 监视器执行的每个请求等同于一个模拟请求。

    # Synthetic actions/requests consumed per monitor = (# Synthetic actions included in monitor) x (# Executions per hour) x (# Locations) x # Hours

    由于用户行为（例如页面加载）不是由用户输入直接触发，而是由模拟浏览器发出的 XHR 或 Fetch 请求不会导致用户动作，因此不会计数。此类 XHR 和 Fetch 调用被视为模拟操作的子请求。

    模拟动作/请求计算示例

    例如，记录的浏览器 clickpath 浏览 2 个页面并单击触发 XHR 或 Fetch 请求的 1 按钮会消耗 3 个模拟动作。如果这样的模拟监视器每隔 15 分钟从 2 个位置运行 1 天，则浏览器 clickpath 将每天消耗 576 个模拟动作。

    3 (synthetic actions) x 4 (monitor executions per hour) x 2 (locations) x 24 (hours per day) = 576 (synthetic actions)

    有关详细信息，请参阅模拟监控。

    数字体验监控超额

    如果您已配置数字体验监控配额（即，您的账户允许您超过 DEM units 的最大限额），您消耗的 units 数与正常 DEM units 消耗一样计算; 每次额外的超额会话或模拟测试都会增加您帐户消耗的 DEM 单位数量。

    日志监控

    Dynatrace SaaS - 年平均日志存储（GiB）

    日志监控消耗基于年平均日志存储大小的预期 GiB，其计算为未压缩日志数据的平均每日提取量乘以天数。达到此限制后，您需要联系销售点击这里以安排额外容量。

    Annual average daily log storage = Actual annual storage / (# of days)

    日志存储计算示例

    例如，假设您的日志监控协议配置为 90 天，并且您已安排每年平均存储量为 450 GiB。在这种情况下，日志数据的预期平均每日摄取量为 5 GiB。

    450 (GiB; base quota of annual average storage) / 90 (days) = 5 (GiB; anticipated average daily ingestion)

    一旦提取和超过年度相当于 1,825 GiB，也可达到 450 GiB 的年平均存储量。

    5 (GiB; anticipated average daily ingestion) x 365 (days) = 1,825 (GiB; anticipated average annual ingestion)

    继续上面的示例，如果六个月后您的实际日志提取量仅为 912.5 GiB（预期的 1,825 GiB 的 50％），那么您可能决定将日志监控分配重新配置为 45 天，同时保留年平均存储量容量不变，为 450 GiB。在这种情况下，预计平均每日提取后续六个月的日志数据将为 10 GiB。

    450 (GiB; average annual capacity) / 45 (days) = 10 (GiB; anticipated average daily ingestion)

    一旦提取年度相当于 2,737.5 GiB，也可达到 450 GiB的年平均存储量。

    (5 x 182.5) + (10 x 182.5) = 2,737.5

    如果您已配置年度日志存储容量，则您的使用将每年重置（仅限 Dynatrace SaaS）。

    日志监控配额 - SaaS 部署

    如果您已配置日志监控超额，以便超出预期年度平均存储大小的约定最大限制，则将根据存储大小限制与实际存储大小之间的差异计算您的超额。例如，如果您同意存储限制为 450 GiB 且实际消耗量为 500 GiB，则超过 50 GiB。

    500 (GiB; actual average storage size) - 450 (GiB; average storage size limit) = 50 (GiB; overages)

    Dynatrace 托管 - 年平均日志存储（GiB）

    日志监控消耗基于年平均日志存储大小的预期 GiB，其计算为未压缩日志数据的平均每日提取量乘以天数。达到此限制后，您需要联系销售 点击这里 以安排额外容量。

    例如，如果在年度期间发送到您的 Dynatrace 托管群集的总日志数据为 730 GiB，那么年平均提取的“每日”大小将为 2 GiB。

    730 (GiB; actual annual ingestion) / 365 (days) = 2 (GiB; annual average daily ingestion)

    如果您已设置年度日志存储容量，您的使用将每年重置。

    日志监控超额 - 托管部署

    如果您已配置日志监控配额，以便超出商定的每日日志存储最大限制，您的超额将根据您的每日存储限制与实际每日存储大小之间的差异计算。例如，如果您达成协议的存储限制为 10 GiB /天且您的实际消耗量为 12 GiB /天，那么您将拥有 2 GiB /天的超额服务。

    12 (GiB; actual daily log storage) - 10 (GiB; daily log storage limit) = 2 (GiB; daily overage)

    自定义指标（Custom metrics）

    开箱即用的 Dynatrace 技术，支持包含有限的自定义指标提取和分析。要配置其他自定义指标提取和分析，请联系 这里。有关在 Dynatrace 中设置和使用自定义指标的完整详细信息，请参阅自定义指标提取和分析。

    每个组件和每个收集的维度值都会使用自定义指标。这里有些例子：

    1. 主动收集 2 个主机的 1 个 File count 指标等同于 2 个自定义指标。
    2. 主动收集跨 10 个主机的 5 个不同的文件夹的 1 个 File count 指标等于 50 个的自定义指标。 5 * 10 = 50 (custom metrics)
    3. 主动收集 5 个不同文件夹（每个文件夹代表一个单独的维度值）的 1 个 File count 指标相当于 5 个自定义指标。 5 * 1 = 5 (custom metrics)

    “主动收集”意味着在 24 小时的滑动窗口内报告自定义指标的至少一个数据点。如果 24 小时或更长时间内没有数据写入，则自定义指标不会计入您的消耗中。

    请注意，对于早于 1.168 的 Dynatrace 版本，滑动窗口定义为 28 天。

    自定义指标超额

    如果您的协议规定了在任何 24 小时内最多可获得 10K 自定义指标，并且在高峰流量期间存在超过此金额的风险，请考虑为自定义指标设置超额。启用超额功能后，在高峰时间您可以超出最大限制。

    例如，假设您在任何 24 小时内达成了限制 100 个自定义指标的协议，您已为自定义指标启用了超额，并且在使用 Dynatrace 的前 15 天，您的环境会提取 90 个报告的自定义指标通过自定义 OneAgent 插件。如果在第 16 天，您的团队启用了一个自定义插件，该插件可以收集额外的 30 个自定义指标（现在总共有 120 个自定义指标），那么您将拥有 20 个自定义指标超额。

    120 (actual custom metrics usage) - 100 (custom metrics limit) = 20 (custom metrics overages)

    免费的自定义指标
