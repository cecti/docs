    Dynatrace 词汇表

    深入了解 Dynatrace 术语和概念

    A

    • Adpex

    性能测量标准，显示记录的性能测量与实际用户满意度之间的关系。Apdex提供统一的方法来衡量性能如何满足用户期望。有关 Apdex 标准的完整详情，请访问 Apdex.org。在 Dynatrace 的上下文中，Apdex 用于为您提供快速简便的评级，您可以使用它来评估应用程序最终用户的满意度。Dynatrace 中的 Apdex 评级基于特定于应用程序的阈值。Apdex 测量等级1相当于完美的性能。Apdex 评级低于0.5等于表现不佳。有关如何在 Dynatrace 中应用 Apdex 的更多详细信息，请参阅什么是 Apdex 评级？

    • Application

    Web 应用程序由 Web 服务器和 Web 容器（例如 Tomcat）提供的 Web 页面组成。Web 或移动应用程序基于处理 Web 请求，Web 服务调用和消息传递等请求的服务构建。在 Dynatrace 中，术语应用程序是指您可以通过浏览器或移动应用程序访问的前端部分。

    Dynatrace 监控您的所有应用程序的服务，流程和基础架构。通过集中评估所有组件，Dynatrace 精确指出每个服务如何为应用程序的性能做出贡献。例如，在 Java 监视中，主机，JVM 和进程被视为一个整体。

    Dynatrace 监视构建应用程序的所有单个组件：Web 请求，数据库请求，进程，服务等。这些单独的组件协同工作，共同提供最终用户作为完整应用程序的视图。Dynatrace 以高度精细的级别识别和跟踪每个应用程序组件。Dynatrace 不是简单地识别服务在进程中的服务器上运行，而是识别这些组件的所有细节。例如，Dynatrace 可以显示您的 Tomcat 应用程序服务器上正在运行哪些 Web 服务，以及 Tomcat 服务器的性能如何受其主机甚至 VMware vCenter 服务器的影响。通过这种方式，Dynatrace 可以让您深入了解独特应用程序堆栈的性能，包括它的所有组件和连接。

    B

    • Brower clickpath

    模拟用户访问，监视应用程序的业务关键工作流。使用 Dynatrace 记录器记录您感兴趣的、用于监视可用性和性能的准确的点击和用户输入序列。一旦您捕获了希望 Brower clickpath 包含的鼠标点击和其他用户操作，您的合成监控器就会定期自动运行，以测试站点的可用性和功能。

    • Brower monitor

    相当于模拟用户使用现代 web 浏览器访问您的应用程序。可以将 Brower monitor 配置为从任何全局位置运行，频率最高为每5分钟一次。当应用程序无法从 Internet 访问或基线性能显著下降时，浏览器监视器会发出警告。

    C

    • Candidate

    请参阅监控候选人。

    • Cluster Management Console (CMC)

    基于 Web 的用户界面，用于管理单个 Dynatrace Managed 群集。CMC允许您：

    • 查看部署状态并配置 Dynatrace Managed 群集。
    • 管理群集节点和 ActiveGates。
    • 管理您的监控环境。
    • 创建用户帐户和组。
    • 查看许可信息和消费状态。

    • 查看审核日志和群集事件。

    • Crash analysis

    确定进程崩溃的根本原因。Dynatrace 通过为每个受影响的进程和受监视的主机提供进程崩溃条目来加速崩溃分析。可用于崩溃分析的详细信息可能包括本机核心转储，Java 核心转储或由于异常而导致的异常程序退出。Dynatrace 还提供了对其他崩溃构件的访问，如hs_err_pid包含的 Java 崩溃文件，文本文件提供 Linux 和 Windows 核心转储分析和文件.NET，Java或Node.js异常有潜在负责崩溃。

    • Cluster

    Dynatrace 监控基础架构。群集包含使用 Dynatrace Managed 安装程序设置的一个或多个群集节点。成功安装后，每个Dynatrace Managed 群集节点都运行 Dynatrace 服务器，ActiveGate 以及 Cassandra，Elasticsearch 和 NGINX 的一个实例。每个群集还提供称为群集管理控制台的基于Web的管理界面。

    • Cluster ActiveGate

    Dynatrace 组件，提供与基于云的 Dynatrace 集群节点的安全可靠通信。它从您的 Dynatrace Managed 群集下载并安装在您的基础结构中。与代理类似，ActiveGate 通过加密和压缩网络通信来降低带宽需求。它还提供对虚拟基础架构的监控。停止或禁用 ActiveGate 会降低使用 Cluster ActiveGate 与 Dynatrace 集群通信的主机的监视功能。

    D

    • Data center

    从 Dynatrace Smartscape 环境 - 拓扑映射的角度来看，数据中心可以是：

    • 在 Amazon 云实例中运行的一组虚拟机。Dynatrace 将可用区域映射到数据中心。
    • 或者一组基于 vCenter 的虚拟主机，通过单个 ActiveGate 将数据传输到 Dynatrace。您可能有多个 vCenter 服务器或与单个数据中心关联的独立 ESXi 主机。

    数据中心内的每个虚拟主机都必须安装 Dynatrace OneAgent。如果安装了 ActiveGate，您将在 Smartscape 视图中看到至少一个数据中心。如果您的环境中没有 ActiveGate，Smartscape 无法为拓扑图创建数据中心。

    请注意，Dynatrace 为数据中心提供真实的地理位置详细信息 - 在 Smartscape 和主机页面（在“ 属性”窗格中）中查找数据中心位置详细信息。

    • Davis Assistant

    Davis Assistant 是 Dynatrace 的对话界面，可让您在聊天应用程序和支持语音的设备中查询常见的与性能相关的问题。Davis 使您能够使用自然语言响应和图表可视化轻松获得有关 Dynatrace 监控数据的可操作见解。支持的聊天应用程序和设备包括 Amazon Alexa，Google 智能助理和 Slack。

    • Deep process monitoring

    Dynatrace 使您可以深入了解构成 Web 应用程序的所有服务的性能：Web 服务，Web 容器，数据库请求，自定义服务等。您甚至可以使用历史数据和最新数据跟踪服务性能。

    • Duration

    合成监视器或合成操作在Web应用程序准备好启动另一个用户操作之前等待的时间。

    E

    • Environment

    请参阅监控环境。

    • Environment ActiveGate

    绑定到您的环境的 ActiveGate，因此仅处理来自属于与 ActiveGate 相同环境的 OneAgent 实例的流量。Environment ActiveGate 从 Dynatrace 环境下载并安装在您的基础结构中。

    Environment ActiveGate 仅处理来自 OneAgent 的流量，而不处理来自其他 ActiveGates 的流量。Environment ActiveGate 可以直接或通过 Cluster ActiveGate 与 Dynatrace 服务器通信。

    • Environment topology

    Dynatrace 监控环境中主机，进程，服务，应用程序和数据中心的 Smartscape 可视化以及这些实体之间的关系。

    在没有手动配置的情况下，Smartscape 可以自动发现整个技术堆栈的端到端所有组件和依赖关系，并在易于理解的环境拓扑图中对其进行可视化。

    • Event

    在您的环境中的服务器上执行的手动操作。Dynatrace 报告事件，并在适用的情况下与检测到的问题相关联。事件示例包括计算机重新启动，系统关闭，进程重新启动和新代码部署。您在服务器上执行的任何手动操作都是一个事件，即使它是一个定期安排的事件。

    • Ecternal service

    您的应用程序或其他未由 Dynatrace OneAgent 直接监视的服务调用的任何服务都被视为外部服务。当 Dynatrace 发送或接收来自受监控服务和应用程序的请求时，它们会保持对外部服务的了解。但是，Dynatrace 不会收到与外部服务相关的性能数据。此规则的一个例外是数据库服务。如果在将 JDBC 调用发送到数据库的计算机上安装了 Dynatrace OneAgent，则 Dynatrace 将提供与被调用数据库服务相关的全方位性能分析（即使数据库服务器计算机本身未安装 Dynatrace OneAgent） 。

    H

    • Host unit

    用于授权目的的主机大小（基于主机提供的 RAM 量）。对于全栈监视，具有16 GB RAM 的主机等同于1个主机单元。对于云基础架构监控，具有16 GB RAM 的主机等于0.3的主机单元。小数主机单元仅适用于 RAM 少于16 GB的主机。

    例如，内存大小为24 GB的主机相当于2.0个全栈监控主机单元和0.6个云基础架构监控主机单元。

    有关完整详细信息，请参阅如何计算监视消耗。

    • Host unit hours

    表示一段时间内主机单元的消耗量。1个主机单位小时相当于1个主机单元消耗1小时。具有16 GB RAM（即1个主机单元）运行一整天的主机消耗24个主机单元小时。

    有关完整详细信息，请参阅如何计算监视消耗。

    • Hyperlyzer

    这个 Dynatrace 多维可视化界面可让您分析应用程序使用情况的前三个发现的多个维度。使用 Hyperlyzer，您可以查看用户所在位置，他们使用的浏览器版本，操作系统以及应用程序收到的不同用户操作数量的详细信息。Hyperlyzer 滚轮界面使您可以快速专注于感兴趣的特定尺寸。

    Hyperlyzer 数据基于检测规则，可以为每个应用程序单独配置。

    K

    • Key request

    关键请求是您的 Web 应用程序环境中的关键请求，因为它们是衡量业务成功的关键指标（例如，登录请求或购物车结帐请求），因为它们提供您的应用程序所依赖的重要的技术功能，或者因为它们涉及复杂的计算。Dynatrace 为此类密钥请求提供高级监控功能，因此您可以重点监控它们。

    Key request具有长期指标历史记录和专用仪表板磁贴，可用于从信息中心进行制图和直接访问。密钥请求始终会被警告，即使它们的吞吐量低于1％。它们还提供自定义阈值。

    密钥请求在每个服务概述页面的密钥请求部分中突出显示。此可见性对于低容量，高重要性请求尤其有用，否则这些请求将显示在服务概述页面的“请求顶部”部分的底部。

    L

    • Log monitoring

    智能评估由网络设备，操作系统，应用程序和其他智能或可编程设备生成的数据记录记录。

    Dynatrace 日志监控提供对所有系统任务关键型流程的日志内容的直接访问，使您能够快速搜索特定的日志消息。您可以根据关键字或分析时间帧过滤日志内容，甚至可以同时分析多个日志文件 - 即使日志文件存储在多个主机上也是如此。

    最重要的是，Dynatrace 人工智能会自动将相关日志消息与您环境中检测到的任何问题相关联。与问题相关的任何相关日志消息都会考虑到该问题的根本原因分析中。

    M

    • Merged services monitoring

    监视来自相同类型和相同进程组的服务的数据。

    合并服务是聚合同一进程组的多个 Web 请求服务的服务，这些服务跨不同的集群节点执行相同的功能。服务合并仅适用于在同一进程组中执行相同功能的 Web 请求服务，因此从性能监视角度来看实际上是相同的。合并服务在 Dynatrace 中显示为包含所有聚合服务的所有数据的单个服务。

    假设你有几个虚拟主机定义的 Apache Web 服务器（例如dynatrace.com，dynatrace.at和dynatrace.pl）。从 Apache 的角度来看，这些是独立的虚拟主机。因此，Dynatrace 将它们检测为单独的服务。但是，出于监视目的，您可能希望将这些服务视为名为Dynatrace web page的单个合并服务。

    为了让您了解哪些服务符合服务合并的要求，Dynatrace 会自动为您识别可合并服务，并在合并服务监控页面上显示它们（设置 > 服务器端服务监控 > 合并服务监控）。只能合并此列表中包含的服务。

    • Mission Control

    对 Dynatrace Managed 的主动支持可自动解决许多常见的维护和支持挑战。Mission Control 的主动支持提供全自动管理功能，使 Dynatrace Server 保持安全，可靠和最新，同时为您节省升级和故障排除等管理任务。

    当您遇到问题时，专门的24x7支持团队会主动处理运行状况检查，扩展验证，升级和任何故障排除。这包括：

    • 分析硬件利用率，并在需要更多资源时提醒您。
    • 解决问题或与 Dynatrace 服务器不兼容 - 因此您无需跟踪和响应系统事件。
    • 优化 Dynatrace Managed 设置以确保最佳性能和稳定性。
    • 收集所需的日志文件以解决问题。

    请参阅 Mission Control 主动支持如何工作？欲获得更多信息。

    • Monitoring candidate

    主机（VMware 虚拟机或 EC2 实例）与您环境中受监控的主机通信，但本身并未安装 Dynatrace OneAgent。建议您在所有监视候选项上安装 Dynatrace OneAgent，以获得完全可见性和完整的监视功能。

    在 Smartscape 中，候选人用虚线圆圈和通用主机图标可视化。候选人显示与相关数据中心相关联。每个监视候选项也列在“ 主机”页面的“未监视”类别中。

    任何不活动的监控候选者（那些未与主机通信超过两小时的候选者）都不包含在 Smartscape 中或列在“ 主机”页面上。

    • Monitoring environment

    包括 Dynatrace 为分析您的应用程序或应用程序提供的所有仪表板，图表，报告和其他工具。监视环境类似于提供所有 Dynatrace 应用程序性能分析功能的分析服务器。

    您可以设置多个监视环境以对相关实体进行分组以进行离散分析。例如，您可以设置一个监视环境来监视和分析生产群集的性能。您可以设置第二个环境，专用于开发人员计算机的性能和第三个用于登台服务器的环境。如何细分监控环境完全取决于您。

    • Milti-dimensional baselining

    有许多影响应用程序性能的变量（或维度） - 最显着的是地理区域，操作系统，浏览器，连接类型和用户操作。这些是 Dynatrace 用于评估应用程序运行状况的五个维度。每个群组的子集都受这些变量的唯一集合的影响，因此Dynatrace分别测量这些用户子集的健康状况并报告每个用户子集。

    O

    • OneAgent

    Dynatrace OneAgent 监控应用程序部署环境中主机的性能以及所有相关进程。根据您的环境设置，Dynatrace OneAgent 可以对您的应用程序及其服务进行详细的性能监控和真实用户监控。对应用程序的更改以及基础结构和性能问题将报告给 Dynatrace 以进行性能监视。

    • OneAgent for Android

    Dynatrace 的Android 移动应用程序支持。使用 Dynatrace 自动检测为您的 Android 应用程序设置用户体验监控，自动将 OneAgent 添加到您的应用程序并对其进行检测而无需修改源代码。

    由于自动检测会在您生成 APK 文件后修改移动应用的字节代码，因此在将已检测的 APK 上传到 Google Play 商店之前，请确保已使用 Google Play 证书对已检测的应用进行签名。OneAgent for Android 可让您跟踪 Android 应用启用 UEM 后由 Android 应用触发的访问者，操作和PurePath 。

    OneAgent for Android 与 AppMon 中的“移动代理”同义。有关详细信息，请参阅 AppMon 文档中的 Mobile UEM。

    • OneAgent for iOS

    Dynatrace 的 iOS 移动应用程序支持，可让您跟踪移动网页上的用户行为。OneAgent for iOS 可帮助您在影响用户之前识别移动用户体验问题。它还允许您接收业务分析数据，例如网页访问者的地理分布和平均会话持续时间。

    Dynatrace 允许您使用 CocoaPods 或 Carthage 设置 OneAgent for iOS，或者您可以通过Dynatrace 手动设置它以启用 iOS 应用程序的 UEM 监控。

    OneAgent for iOS 与 AppMon 中的“移动代理”同义。有关详细信息，请参阅 AppMon 文档中的 Mobile UEM。

    • oneagentmon device

    在 Dynatrace OneAgent 安装期间，“oneagentmon device”出现在 Windows 系统中。它被 Dynatrace 用于深度过程监控。它像监视驱动程序一样工作，允许 Dynatrace OneAgent 在操作系统和它监视的进程之间添加自己的库。

    • oneagentmon dirver

    在 Dynatrace OneAgent 安装期间出现在 Windows 系统中的特殊设备。它对于深度监视而言是必不可少的，因为它允许 Dynatrace OneAgent 在系统和受监视进程之间添加自己的库。

    • Opaque service

    由 Dynatrace 在主叫端检测到的服务，其代码级可见性不可用。

    Dynatrace 可以检测不透明服务的请求并识别它们处理的进程，但 Dynatrace 无法直接监视不透明服务。

    P

    • Prediction-based anomaly detection

    评估流量和负载异常与评估性能异常相比具有不同的假设。Dynatrace 使用一系列度量和方法来识别影响客户体验的性能异常，因此需要您的关注。但是，流量和负载异常是基于应用程序的业务模型，相关营销工作和社会因素驱动的日常，季节性和业务周期相关模式。这些是相当可预测的周期，当客户使用您的应用程序时会影响流量和负载。可预测周期的示例包括周末/工作周，工作日/晚上时间以及在诸如黑色星期五之类的年度周期中发生的客户活动。因此，Dynatrace 使用基于预测的方法来检测应用程序流量和服务负载的异常。

    • Problem

    Dynatrace 中的信息逻辑分组，包括 AI 驱动的分析，环境上下文，根本原因分析以及为您环境中的一个或多个事件提供的其他详细信息。Problem 可能会在您的环境中表现为性能下降，功能不正常或缺乏可用性。问题可能是单个“事件”或多个事件的结果。

    • Process

    进程是执行的计算机程序的实例。进程充当托管服务的容器。当您查看流程时，您会看到拓扑信息，而服务则会为您提供代码级别的洞察力。例如，您可能有一个 Tomcat 进程以服务器端服务的形式托管 Web 应用程序。虽然进程以主机为中心与环境中的单台计算机相关联，但服务是以请求为中心的，因此通常跨越数据中心内的多台计算机。

    • Process group

    一组在多个主机上执行相同功能的进程。例如，您可能拥有一组服务器，每台服务器运行相同的进程以支持多个主机。

    • Process responsiveness

    基于响应时间度量的流程运行状况评估，用于衡量对来自其他进程或客户端的请求做出响应所需的时间（无论资源消耗如何）。

    R

    • Real user monitoring

    真实用户监控可分析客户在与应用程序交互时的体验。真实用户监控是网络分析和性能测量的结合，可以构建面向客户的组件的性能表现。

    • Response time

    服务执行和完成任务所需的时间。

    • Root cause analysis

    Dynatrace AI 驱动的流程，用于了解发现问题的原因，以及解决问题根本原因的指导。

    Dynatrace 会在客户受到影响之前识别性能问题，并根据客户影响确定问题的优先级，从而即时了解问题严重性和用户体验影响。当 Dynatrace 发现性能问题时，它会为您提供单个问题通知，以确定问题的根本原因，因此您无需手动解释数十个数据源以确定需要修复的内容。

    由于性能问题很少是孤立事件（它们通常是较大问题的症状），因此 Dynatrace 人工智能可以分析 IT 环境中发生的数十亿事件，以帮助您解决问题的原因，而不是症状。Root cause analysis 还通过深入分析源代码和数据库语句简化了错误修复，使您可以将平均修复时间缩短90％或更多。

    S

    • Server-side service

    Web 和移动应用程序基于处理 Web 请求，Web 服务调用和消息传递等请求的服务构建。例如，发送到特定 Tomcat 服务器的 Web 请求是服务器端服务的示例。此类服务可以采用 Web 服务，Web 容器，数据库请求，自定义服务等形式。服务可以依次调用其他服务，例如Web服务，远程服务和数据库服务。

    进程本质上是托管服务的容器。当您查看正在查看拓扑信息的流程时，服务可以提供代码级别的洞察。例如，您可能有一个 Tomcat 进程以服务器端服务的形式托管 Web 应用程序。虽然进程以主机为中心与环境中的单台计算机相关联，但服务是以请求为中心的，因此通常跨越数据中心内的多台计算机。

    • Service

    在进程组中执行的服务器端代码。

    • Service backtrace

    可视化发送请求到特定后端服务的所有服务。服务回溯不仅仅显示哪些服务直接调用特定的后端服务，而是向您显示导致每个请求的服务调用序列，一直回到触发调用序列的浏览器点击。

    • Service quality report

    这是一份易于阅读的报告，总结了 Dynatrace 过去一周编制的监控见解。服务质量报告的结构使得即使是非技术团队成员也能理解它们。报告会计算受监控组件的执行情况，并为以下区域提供分数：

    • 整体环境
    • 应用
    • 服务
    • 基础设施

    服务质量报告可让您深入了解环境中的热点，并轻松与他人分享见解。

    • Smartscape

    使用专有的 Smartscape 技术，Dynatrace 能够自动发现整个应用程序堆栈，包括构建基础架构和网络的数据中心，主机和进程，构成应用程序的服务，并一直扩展到应用程序。Dynatrace 甚至可以实时发现并映射这些组件的所有依赖关系。然后，Dynatrace 将在交互式 Smartscape 拓扑图中向您展示其发现。您可以单击 Smartscape 可视化中的任意位置，以查看环境中每个已发现实体的性能统计信息。

    • Synthetic action

    相当于模拟用户操作。当浏览器单击路径触发加载新页面或XHR请求时，Dynatrace会监视该操作的可用性和性能。

    • Synthetic monitoring

    在任何给定时间，全世界的不同客户都可以以不同的方式体验应用程序可用性和性能。Dynatrace 综合监控使您可以轻松监控应用程序的可用性和性能，因为客户可以在全球范围内全天候体验它。即使当前没有用户访问您的站点，综合监控也会主动模拟用户访问。综合监控通过推动具有完整 HTML5 / Ajax 支持的真实 Web 浏览器会话，为您的 Web 应用程序提供24x7全天候的可视性。

    T

    • timeframe selector

    分析时间范围选择工具位于所有 Dynatrace 页面和视图的右上角。时间范围选择器使您可以将监控数据过滤到特定时间范围。上一页和下一页按键可以基于预定义的增量（以向前和向后移动时间30 minutes，1 hour，1 day，1 week，等等）。即使在页面和视图之间移动，选定的时间范围仍然保持不变。

    U

    • User action

    客户在您的应用程序中执行的操作。用户操作等同于常见的用户活动，例如执行搜索，查看帐户余额，查看购物车中的商品或订购产品。用户操作因应用程序的功能而异。

    从 Dynatrace 的角度来看，每个用户操作都包含从一个状态转换到另一个状态期间发生的所有客户端活动。每个用户操作都包含至少一个 Web 请求，因为当用户导航到页面时，浏览器会下载多个资源并调用 JavaScript onload 处理程序，而后者可能会执行 XHR 请求等等。

    用户操作的持续时间称为操作持续时间。这表示用户在继续操作之前必须等待的时间。因此，低动作持续时间优于高动作持续时间。有关更多详细信息，请参阅什么是用户操作？。

    • User session

    一个或多个用户操作，包含用户浏览器和应用程序之间的交互。用户会话（又名“用户访问”）在用户的浏览器关闭或30分钟不活动时结束。请注意，退回会话（仅包含单个操作的会话）不会计入您的许可会话配额。

    Dynatrace 通常显示每个用户的所有会话，即使这些会话是匿名的（使用浏览器 cookie 标识的未经过身份验证的用户），或者已编辑或删除用户标记。对于移动应用程序，Dynatrace 会根据用户使用的特定移动设备识别各个用户。对于 Web 应用程序，通过在每个用户的浏览器中存储持久 cookie 来实现用户识别。Cookie 使 Dynatrace 能够为已知用户分配匿名用户会话。只要用户至少登录过您的应用程序一次，您就可以搜索并识别该用户，即使用户在匿名的未经身份验证的会话中访问您的应用程序也是如此。
